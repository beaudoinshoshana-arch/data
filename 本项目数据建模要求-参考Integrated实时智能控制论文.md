# 本项目数据建模要求

副标题：参考《Integrated real-time intelligent control for wastewater treatment plants: Data-driven modeling for enhanced prediction and regulatory strategies》

## 1. 目标定位

本项目的数据建模必须服务于“曝气 + 加药”智能决策，而不是只做静态预测。参考论文的方法，模型体系应同时满足以下三类目标：

1. 用尽量少的关键在线变量，预测未来一小时关键出水指标变化。
2. 将预测模型作为代理模型，支撑曝气强度与药剂投加量的约束优化。
3. 预留动态更新能力，使模型可以随着工况变化周期性刷新，而不是一次训练后长期固定。

结合当前项目数据基础，建模主任务定义为：

- 预测目标：下一小时 `effluent_cod_mgL`、`effluent_nh3n_mgL`、`effluent_tp_mgL`、`effluent_tn_mgL`
- 决策目标：输出 `aeration_intensity_pct` 与 `chemical_dose_pac_mgL` / `chemical_dose_kgph` 的推荐值
- 建模原则：少变量、强时序、可更新、可解释、可直接接入现有 `stage1 -> stage2` 流水线

## 2. 参考论文对本项目的可落地映射

论文的核心不是单纯换一个算法，而是把“关键底物的动态特征”提取出来，再进行预测和控制。本项目应按下面的映射关系落地：

| 论文设计 | 本项目落地要求 |
| --- | --- |
| 少量关键底物在线监测驱动模型 | 以进水 COD、BOD、NH3-N、TP、流量，过程 DO、MLSS，以及当前出水指标作为主输入 |
| 二阶导数近似高斯分布，提取 `A / μ / δ` 动态特征 | 当前小时级数据先做“代理版动态特征”，后续有 10 s / 5 min 数据后再升级为论文同款特征参数函数 |
| 控制变量为 `KLa` 和 `NRR` | 本项目当前控制变量映射为 `aeration_intensity_pct` 和 `chemical_dose_pac_mgL`；若未来新增回流比或硝化液回流流量，再扩展为三变量优化 |
| 3 h 更新模型、30 min 执行控制 | 当前版本按小时数据运行“每到新样本重新推理”；增强版再实现“3 h 参数刷新 + 30 min 控制滚动” |
| 批式模拟器 + 机理模型支撑动态特征学习 | 本项目先不强制引入 ASM2D 复刻；以现有真实数据 + 场景扩增数据完成基线，后续再补机理仿真层 |
| 四分位数阈值过滤异常 | 沿用现有工程阈值、Hampel、平直段检测，并补充自适应分位数规则 |

## 3. 当前项目的数据基础与约束

根据现有产物，当前项目已经具备可直接建模的数据底座：

1. `plant_real_hourly.csv`：4574 条小时级单厂主表，时间范围为 `2025-01-01 00:00:00` 到 `2025-07-10 13:00:00`
2. `decision_dataset_raw.csv`：13635 条决策样本，包含 `observed`、`load_up`、`rain_dilution` 三类场景
3. `decision_labels_observed.csv`：观测场景的未来一小时出水标签
4. `public_monitor_long.csv`：公开监测长表，可用于补充分布边界、异常阈值和场景库，不应直接与单厂监督标签混训

因此，本项目的数据建模要求必须遵守以下现实约束：

1. 当前主训练粒度以“小时级”作为统一标准，不得假设已经具备论文中的 10 s 批式反应器数据。
2. `observed` 场景用于监督学习，`load_up` 与 `rain_dilution` 主要用于推荐搜索和鲁棒性测试，不直接用于拟合标签。
3. 扩展变量如 `reactor_orp_mv`、`reactor_no3_mgL`、`internal_nh3_mgL` 可以纳入建模，但不能因为单个传感器质量不稳定而阻塞基线模型上线。
4. 公开数据只能作为外部先验、范围约束、异常参考和补充分析，不得在未做厂站域适配前直接与本厂标签混合训练。

## 4. 数据建模的硬性要求

### 4.1 数据层结构要求

建模数据必须拆成四层，而不是只有一张训练表：

1. 原始层：保留多源原始时间序列与原始单位，不覆盖原始值。
2. 对齐层：形成统一时间索引的单厂主表，当前标准粒度为 1 h。
3. 特征层：保存可复用的监督建模特征、动态特征、质量标记和场景标记。
4. 决策层：保存推荐前基线值、候选动作、预测结果、目标函数值和最终推荐动作。

最低必备主键字段：

- `timestamp`
- `source_flag`
- `split`
- `scenario_tag`

最低必备输入字段：

- `influent_cod_mgL`
- `influent_bod_mgL`
- `influent_nh3n_mgL`
- `influent_tp_mgL`
- `influent_flow_m3h`
- `reactor_do_mgL`
- `sludge_mlss_mgL`
- `aeration_intensity_pct`
- `chemical_dose_pac_mgL`
- `chemical_dose_kgph`
- `effluent_cod_mgL`
- `effluent_nh3n_mgL`
- `effluent_tp_mgL`
- `effluent_tn_mgL`

推荐扩展字段：

- `influent_tn_mgL`
- `reactor_orp_mv`
- `reactor_no3_mgL`
- `internal_nh3_mgL`
- `effluent_flow_m3h`
- `effluent_ph`
- `effluent_temp_c`

### 4.2 时间粒度与对齐要求

1. 当前正式建模口径统一为小时级，所有训练样本必须对齐到整点时间索引。
2. 若未来接入分钟级或秒级在线传感器，必须同时保留原始高频表与小时聚合表，不能只保留聚合结果。
3. 推荐推理时点必须与可用输入数据严格对齐，不允许使用未来信息回填当前特征。
4. 未来若实现论文式动态更新，必须新增：
   - `30 min` 控制快照表
   - `3 h` 动态特征更新窗口表
   - 高频传感器原始缓存表

### 4.3 数据清洗与质量控制要求

数据预处理必须保留“工程规则 + 统计规则”双重机制：

1. 工程阈值过滤：按各指标合理物理范围过滤负值与超量程值。
2. 自适应分位数过滤：参考论文的 quartile 思路，对异常点进行分位数阈值识别。
3. Hampel 异常检测：用于识别短时尖峰噪声。
4. 平直段识别：用于识别传感器卡死或长时间不变值。
5. 缺失处理：短缺口插值，中等缺口滚动中位数或历史同小时中位数回填，长缺口剔除出监督训练。
6. 质量标志保留：每个关键字段必须保留异常、缺失、替换、平直段等标志位，供建模时选择是否屏蔽。

特别要求：

1. 当前 `reactor_orp_mv` 质量波动较大，应作为“可选增强特征”，而不是基线必选特征。
2. 所有标签列必须和对应时点特征分开存储，避免特征泄漏。
3. 不允许在训练前删除质量标记列；是否使用这些标记应由实验结果决定。

### 4.4 样本构建要求

监督学习样本必须满足以下规则：

1. 标签采用“未来一小时出水值”，即 next-hour 监督口径。
2. 训练集、验证集、测试集必须按时间顺序切分，禁止随机打乱。
3. 只有 `scenario_tag == observed` 的样本可参与监督拟合。
4. `load_up`、`rain_dilution` 场景必须保留在决策数据集里，用于候选动作搜索与鲁棒性评估。
5. 所有建模实验都必须可追溯到具体的 `timestamp` 和 `split`，以便复盘。

## 5. 特征建模要求

### 5.1 基线版特征工程要求

在当前小时级数据条件下，必须先完成一版可运行的“代理版动态特征”，至少包括：

1. 当前状态特征：
   - 进水水质
   - 进水流量
   - 过程状态
   - 当前控制量
   - 当前出水指标
2. 时间周期特征：
   - 小时正余弦
   - 星期正余弦
3. 滞后特征：
   - `1 h`
   - `3 h`
   - `6 h`
4. 滚动特征：
   - `3 h` 滚动均值
   - `6 h` 滚动均值
   - 必要时增加滚动波动率
5. 工程派生特征：
   - 污染负荷
   - 曝气/流量比
   - 投药/流量比
   - 投药/负荷比
   - 关键浓度比值
6. 代理动态特征：
   - 一阶差分斜率
   - 二阶差分加速度
   - 近 3 h 至 6 h 趋势转折指标
   - 异常波动强度指标

### 5.2 论文式动态特征模块要求

当项目具备批式反应器数据或高频在线数据后，必须升级为论文风格的动态特征提取模块。升级版要求如下：

1. 对关键底物时间序列计算一阶、二阶导数。
2. 以二阶导数近似高斯分布为前提，对每个关键底物提取 `A`、`μ`、`δ` 三类参数。
3. 最少形成 9 个动态特征参数：
   - `A_AN_NO3`、`mu_AN_NO3`、`delta_AN_NO3`
   - `A_OX_BOD`、`mu_OX_BOD`、`delta_OX_BOD`
   - `A_OX_NH3`、`mu_OX_NH3`、`delta_OX_NH3`
4. 基于输入水质和控制变量，对上述特征参数建立函数映射关系。
5. 保留论文中的修正因子 `gamma` 思路，将其作为少量可校准参数，而不是重新引入大批不可解释自由参数。

落地约束：

1. 现阶段没有高频批式数据时，不强制在本轮交付中复刻该模块。
2. 现阶段必须先用“差分 + 滚动 + 变化率”形成论文思路的等价近似版本。
3. 一旦新增分钟级或秒级监测数据，动态特征模块应优先于深度学习模型扩展，因为它更符合当前项目的低成本部署目标。

## 6. 预测模型要求

### 6.1 任务定义

预测模型必须以“多输出代理模型”为核心，支持同时预测：

1. `label_next_effluent_cod_mgL`
2. `label_next_effluent_nh3n_mgL`
3. `label_next_effluent_tp_mgL`
4. `label_next_effluent_tn_mgL`

### 6.2 模型族要求

模型实施遵循“两阶段推进”：

1. 第一阶段必须完成树模型基线：
   - `RandomForestRegressor`
   - `ExtraTreesRegressor`
2. 第二阶段可扩展时序模型：
   - LSTM
   - CNN-LSTM
   - Transformer

扩展条件：

1. 若仍以小时级小样本为主，树模型优先，深度学习仅作为补充实验。
2. 若未来接入高频传感器并形成稳定长序列，再将深度学习模型纳入正式候选集。

### 6.3 训练与选择要求

1. 候选模型只能在训练集上拟合，在验证集上选型。
2. 最终模型可在 `train + val` 上重训，并在测试集上一次性评估。
3. 模型选优指标必须使用“污染物重要性加权”的综合指标，不能只看单一目标的 MAE。
4. 模型输出必须附带特征重要性或解释信息，至少支持树模型重要性排序。

特别要求：

1. 若某一目标变量方差极低，`R²` 失真时必须同时报告 `MAE / RMSE / 达标率`，不能只用 `R²` 判定模型失效。
2. `TP` 目标需单独监控，因为低波动情况下容易出现 `R²` 不稳定但绝对误差很小的情况。

## 7. 决策优化模型要求

### 7.1 决策变量

当前版本的优化器必须至少支持：

1. `aeration_intensity_pct`
2. `chemical_dose_pac_mgL`
3. `chemical_dose_kgph`

说明：

1. `chemical_dose_pac_mgL` 和 `chemical_dose_kgph` 本质上是一组联动变量，至少保留一个主优化口径，另一个由流量换算。
2. 若未来新增回流比、硝化液回流流量或风机频率，应升级为多变量约束优化。

### 7.2 目标函数

优化目标必须体现论文中的“出水合规 + 成本最小 + 调节平滑”思想。建议采用统一目标函数：

```text
Min J = w1 * 出水风险 + w2 * 曝气能耗 + w3 * 药耗成本 + w4 * 调节平滑惩罚
```

具体要求：

1. 出水风险由四个预测目标按权重聚合。
2. 曝气能耗与 `aeration_intensity_pct` 正相关。
3. 药耗成本与 `chemical_dose_kgph` 正相关。
4. 平滑惩罚按推荐值相对于当前值的变动幅度计算。

### 7.3 约束条件

约束必须配置化，不能把论文中的排放阈值直接硬编码为唯一标准。系统至少要支持：

1. 出水 COD、NH3-N、TP、TN 的可配置达标阈值
2. 曝气强度上下限
3. 投药量上下限
4. 单次调节最大步长
5. 可选的 DO、MLSS 安全边界

### 7.4 搜索策略

1. 当前版本必须采用局部有界搜索，搜索范围围绕当前动作值展开。
2. `observed`、`load_up`、`rain_dilution` 三类场景必须使用不同的搜索偏移策略。
3. 若未来引入 NSGA-II 或其他多目标进化算法，仍需保留局部搜索基线，便于比较实时性。
4. 推荐结果中必须同时输出：
   - 基线动作
   - 推荐动作
   - 预测改善值
   - 目标函数改善值

## 8. 动态更新与在线运行要求

参考论文，本项目的运行机制应拆成“模型刷新周期”和“控制执行周期”两套节奏：

### 8.1 当前可交付版本

1. 以小时级数据为基础，每来一条新样本重新推理一次。
2. 推荐周期默认按小时执行。
3. 模型刷新可按天、按周或按性能漂移触发。
4. 当最近窗口误差持续恶化时，必须支持再训练或参数回标。

### 8.2 增强版本

1. 新增高频传感器后，动态特征参数函数每 `3 h` 刷新一次。
2. 控制优化每 `30 min` 滚动执行一次。
3. 高频数据预处理必须在进入模型前完成异常抑制与缓存。
4. 模型更新日志、推荐日志、执行反馈日志必须形成闭环。

## 9. 评估与验收要求

### 9.1 预测效果指标

每个目标变量都必须输出：

1. `MAE`
2. `RMSE`
3. `R²`
4. 归一化误差
5. 达标率相关指标

### 9.2 决策效果指标

推荐系统必须至少评估：

1. 推荐前后预测出水风险变化
2. 曝气能耗变化
3. 药耗变化
4. 控制动作波动幅度
5. 各场景下的稳健性

### 9.3 工程可用性指标

1. 推理延迟必须可统计。
2. 推荐结果必须可解释。
3. 输出结果必须保留时间戳和数据版本。
4. 必须可以复现训练、预测、推荐三个阶段的结果。

## 10. 本项目的最小可行建模交付物

本轮项目交付至少应包括以下文件或等价产物：

1. 小时级单厂主表
2. 决策样本表与观测标签表
3. 特征列清单
4. 模型排行榜与最佳模型摘要
5. 测试集预测结果表
6. 测试集推荐结果表
7. 建模总结 `summary.json`
8. 一份对外可读的模型说明文档

建议新增但不强制的产物：

1. `dynamic_feature_frame.csv`：保存代理动态特征或高频动态特征
2. `constraint_config.json`：保存排放阈值、控制上下限和步长约束
3. `model_card.md`：保存模型输入、输出、假设、限制和适用场景

## 11. 结论

如果严格参考论文思想，本项目的数据建模不应停留在“多做几个滞后特征 + 训练一个回归器”，而应建设成一个分层体系：

1. 底层做关键变量的动态信息抽取
2. 中层做未来出水代理预测
3. 上层做约束优化推荐与周期更新

在当前数据条件下，建议先完成“小时级代理动态特征 + 树模型多输出预测 + 局部有界优化”这一版可运行系统；后续随着高频传感器和回流控制变量接入，再升级为论文风格的高斯动态特征参数模型和更高频的在线控制体系。
