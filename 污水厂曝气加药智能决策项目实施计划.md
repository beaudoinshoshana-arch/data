# 污水厂曝气加药智能决策项目实施计划

## 1. 项目概述

### 1.1 项目名称
污水厂曝气加药智能决策智能体系统

### 1.2 项目背景
根据 [数据要素素质大赛.docx](/D:/part3data/数据要素素质大赛.docx:1) 的赛题要求，本项目面向污水处理厂曝气与加药环节，构建集多源数据接入、数据治理、智能预测、决策优化、可视化展示和运行反馈于一体的智能决策系统。项目目标是在进水水质水量波动、复杂工况变化和节能减耗约束下，实现曝气强度与药剂投加量的实时、精准、可解释决策。

### 1.3 项目总体目标
1. 构建不少于 1 万条、覆盖关键运行参数的污水厂时序决策数据集。
2. 建立可同时服务于预测与控制的污水处理智能建模框架。
3. 实现面向曝气强度与药剂投加量的动态推荐与优化决策。
4. 搭建支持数据接入、分析、推理、反馈和可视化展示的原型系统。
5. 形成可用于参赛报告、系统演示、代码交付和答辩展示的完整项目材料。

## 2. 赛题需求拆解

### 2.1 数据层需求
赛题明确要求采集并构建不少于 1 万条的污水厂运行时序数据，需覆盖以下八个核心参数：

| 类别 | 参数名称 | 建议字段名 | 说明 |
| --- | --- | --- | --- |
| 进水指标 | COD | `influent_cod_mgL` | 化学需氧量 |
| 进水指标 | BOD | `influent_bod_mgL` | 五日生化需氧量 |
| 进水指标 | 氨氮 | `influent_nh3n_mgL` | 氨氮浓度 |
| 进水指标 | 总磷 | `influent_tp_mgL` | 总磷浓度 |
| 过程状态 | 溶解氧 | `reactor_do_mgL` | 生化池溶解氧 |
| 过程状态 | 污泥浓度 | `sludge_mlss_mgL` | 混合液悬浮固体浓度 |
| 控制变量 | 曝气强度 | `aeration_intensity_pct` | 风量或曝气强度标准化表达 |
| 控制变量 | 药剂投加量 | `chemical_dose_kgph` | 药剂质量流量 |

同时，需要保留以下扩展参数用于增强建模能力：
1. `influent_flow_m3h`：进水流量。
2. `influent_tn_mgL`：总氮。
3. `reactor_orp_mv`：氧化还原电位。
4. `reactor_no3_mgL`：硝态氮。
5. `internal_nh3_mgL`：池内氨氮。
6. `effluent_cod_mgL / effluent_nh3n_mgL / effluent_tp_mgL / effluent_tn_mgL`：出水指标。
7. `chemical_dose_pac_mgL`：按浓度口径表示的投药量。

### 2.2 算法层需求
赛题要求融合机器学习、深度学习和强化学习等方法，实现以下任务：
1. 对进水波动和复杂工况进行动态适应。
2. 对曝气与加药参数进行实时决策。
3. 同时兼顾水质达标、能耗降低和药耗降低。
4. 支撑测试集 MAE、RMSE、达标率、节能率等评估指标。

### 2.3 系统层需求
系统需要至少包含以下模块：
1. 多源数据接入模块。
2. 数据预处理与规范化存储模块。
3. 智能体推理与决策模块。
4. 运行效果反馈模块。
5. 统计分析与可视化展示模块。
6. 与现有自控系统的接口适配模块。

## 3. 项目技术路线

### 3.1 总体路线
本项目建议采用“机理启发 + 数据驱动 + 约束优化 + 智能体编排”的路线：

1. 数据治理层：
   对原始监测数据、公开环保数据和模拟场景数据进行清洗、异常检测、缺失填补、时间对齐和统一存储。
2. 过程建模层：
   基于污水处理机理参数与历史运行数据，建立进水变化到出水结果的时序预测模型。
3. 决策优化层：
   以预测模型作为代理模型，联合能耗、药耗、水质约束构建多目标优化决策器。
4. 智能体应用层：
   将数据分析、模型推理、结果解释、报告生成和异常预警整合为智能体工作流。
5. 系统展示层：
   通过 Web 可视化大屏呈现实时监测、趋势分析、控制建议和运行反馈。

### 3.2 推荐算法组合
为兼顾可实现性、效果和答辩展示，建议分层实施：

| 层级 | 算法 | 作用 | 本项目建议 |
| --- | --- | --- | --- |
| 基线模型 | 随机森林、梯度提升树 | 快速建立可解释基线 | 必做 |
| 时序模型 | LSTM、CNN-LSTM、Transformer | 建模时序依赖与突变响应 | 建议至少实现 1-2 个 |
| 决策优化 | 网格搜索、PSO、MPC、NSGA-II | 输出曝气与加药最优组合 | 必做 |
| 智能控制增强 | 强化学习、多智能体 | 在线自学习与复杂场景优化 | 作为扩展项 |

## 4. 数据实施方案

### 4.1 数据来源
1. 污水厂本地运行监测数据。
2. 市政环保部门公开监测数据。
3. 可复现的场景模拟数据。
4. 文献中的典型机理参数和工况规则。

### 4.2 数据处理流程
1. 数据采集：
   Excel、CSV、API、公开网页和 RDF 数据统一接入。
2. 时间对齐：
   按小时或 5 分钟粒度对齐，必要时保留多尺度数据。
3. 异常处理：
   工程阈值过滤、Hampel 检测、平直段识别。
4. 缺失处理：
   短缺口插值、中等缺口历史中位数回填、长缺口剔除。
5. 特征构建：
   构造滞后特征、滚动均值、污染负荷、投药比、气水比、出水趋势等特征。
6. 标准化与存储：
   统一字段、单位、时间索引和数据字典，保存为模型训练表和系统推理表。

### 4.3 数据集目标
1. 原始样本规模不少于 1 万条。
2. 至少形成训练集、验证集、测试集三部分。
3. 形成“观测场景 + 负荷升高场景 + 雨季稀释场景”等决策数据集。

## 5. 模型实施方案

### 5.1 第一阶段：预测模型
目标是预测未来一段时间的关键出水指标，作为后续决策的代理模型。

输入建议包括：
1. 进水指标：COD、BOD、氨氮、总磷、总氮、流量。
2. 过程状态：DO、MLSS、ORP、NO3-N、池内 NH3-N。
3. 控制量：曝气强度、PAC 投加量、回流比。
4. 时间特征：小时、星期、节假日、季节特征。
5. 历史特征：1h、3h、6h、12h、24h 滞后与滚动统计。

预测输出建议包括：
1. 下一时刻或下一小时出水 COD。
2. 下一时刻或下一小时出水 NH3-N。
3. 下一时刻或下一小时出水 TP。
4. 下一时刻或下一小时出水 TN。

### 5.2 第二阶段：决策优化模型
目标是在满足出水达标约束的前提下，输出曝气和加药的推荐值。

优化变量：
1. 曝气强度或风量。
2. 药剂投加量。

约束条件：
1. 出水水质必须满足排放标准。
2. DO、MLSS 等过程状态不能越过安全边界。
3. 曝气和投药调整幅度不能过大。

优化目标函数建议：
1. 最小化出水超标风险。
2. 最小化曝气能耗。
3. 最小化药剂消耗。
4. 最小化控制变量波动。

综合形式可写为：

```text
Min  J = w1 * 出水风险 + w2 * 能耗成本 + w3 * 药耗成本 + w4 * 调控波动惩罚
```

### 5.3 第三阶段：智能体决策编排
智能体建议拆分为以下角色：
1. 数据智能体：负责接入、校验、预处理。
2. 预测智能体：负责未来水质预测。
3. 优化智能体：负责控制量搜索与推荐。
4. 报告智能体：负责生成运行分析与可视化说明。
5. 预警智能体：负责识别异常工况和设备风险。

## 6. 系统实施方案

### 6.1 系统架构
建议采用“数据层 + 算法层 + 服务层 + 展示层”的四层架构。

1. 数据层：
   MySQL / CSV / Parquet 统一存储，保留原始表、中间表、特征表、决策表。
2. 算法层：
   Python 实现预处理、训练、推理和优化。
3. 服务层：
   FastAPI 或 Flask 提供推理接口和数据接口。
4. 展示层：
   Vue、React 或 Streamlit 构建监控大屏和报告页面。

### 6.2 核心页面
1. 实时监测看板：
   展示进水、过程状态、控制参数、出水质量。
2. 智能决策页面：
   展示当前推荐曝气量、推荐投药量、推荐原因和预期收益。
3. 趋势分析页面：
   展示 COD、NH3-N、TP、TN、DO、MLSS、能耗和药耗变化趋势。
4. 场景模拟页面：
   展示负荷突变、雨季稀释等场景下的模型响应。
5. 报告输出页面：
   自动生成运行分析、结果解释和优化建议。

## 7. 评价指标与验收标准

### 7.1 建模指标
1. MAE。
2. RMSE。
3. R² 或 NSE。
4. 达标率。
5. 超标预警命中率。

### 7.2 决策指标
1. 水质达标率不低于 95%。
2. 相较传统经验规则，能耗或药耗降低不低于 10%。
3. 单次决策响应时间尽量控制在 1 秒以内。
4. 在负荷突变、雨季稀释和噪声场景下保持稳定输出。

### 7.3 系统指标
1. 支持多源数据格式接入。
2. 具备可视化展示和交互说明。
3. 具备模型训练、推理、报告生成的完整流程。
4. 具备部署说明和对接说明。

## 10. 风险与对策

| 风险 | 表现 | 对策 |
| --- | --- | --- |
| 数据缺失多 | 样本不足或偏差大 | 引入公开数据与场景模拟补充 |
| 参数分布不稳 | 预测偏差增大 | 做分场景建模和滑窗更新 |
| 决策不可解释 | 难以答辩和落地 | 增加规则约束、特征贡献说明和案例解释 |
| 实时性不足 | 响应超过 1 秒 | 使用轻量模型推理，优化搜索范围 |
| 系统展示不足 | 影响评分 | 强化可视化页面和动态演示材料 |

## 11. 当前建议的最小可行版本

结合当前数据与实现进度，建议先完成以下最小可行版本：

1. 完成 8 个核心参数的数据集构建与清洗。
2. 建立一个树模型基线和一个 LSTM 时序模型。
3. 建立一个面向曝气与加药的局部搜索优化器。
4. 建立一个简单可视化界面展示实时输入、预测结果和推荐输出。
5. 输出一份包含指标、案例和经济效益分析的完整参赛报告。

## 12. 结论

本计划严格对齐赛题要求，并兼顾当前数据基础与本科生项目实现难度。建议采用“先完成可运行、可解释、可展示的基线系统，再逐步增强深度学习与智能体功能”的策略，这样更容易在规定时间内形成稳定成果，并满足比赛对规范性、模型效果、文档可读性和展示丰富性的综合要求。
