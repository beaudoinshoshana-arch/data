# 分步反思记录

## 1. 赛题与现有产物梳理

- 结果：确认赛题要求覆盖数据采集、智能体模型、实时决策、可视化大屏、报告和系统对接。
- 风险：项目容易退化成单纯预测模型，无法体现智能体和控制闭环。
- 改进：用“监督代理模型 + Safe-MARL + 约束盾 + 专家回退”组织系统。

## 2. 数据融合

- 结果：生成融合数据注册表和场景库，本地单厂与公开监测数据统一到长表规范。
- 风险：外部数据字段不一定包含控制变量，直接混训会引入域偏移。
- 改进：外部数据默认用于分布边界、鲁棒性和场景扩增，只有字段可靠时才进入监督训练。

## 3. Safe-MARL

- 结果：实现曝气 agent 与投药 agent 的轻量 PyTorch 策略，所有动作通过安全盾。
- 风险：离线 reward 使用代理响应系数，不能替代现场闭环试验。
- 改进：保留局部专家搜索作为安全对照，并在大屏中暴露可行率、回退率和 reward 分解。

## 4. 后端 API

- 结果：FastAPI 提供摘要、时序、推荐、单点推理、RL 推荐和报告摘要接口。
- 风险：大规模 CSV 首次读取有延迟。
- 改进：使用缓存读取；后续可切换到 DuckDB/SQLite 做大数据查询。

## 5. 前端大屏

- 结果：React + ECharts 大屏展示实时 KPI、预测曲线、动作对比、reward 雷达图和数据源状态。
- 风险：ECharts 首包较大，移动端性能需关注。
- 改进：已配置 manual chunks；后续可按视图懒加载图表。

## 6. 静态托管与作品打包

- 结果：FastAPI 在前端构建后可直接托管大屏首页，`package_submission.py` 生成不含原始大文件的参赛提交包。
- 风险：运行中的旧后端进程不会自动加载新增静态托管逻辑，需要重启后端。
- 改进：新增 `run_dashboard.ps1` 和 `run_pipeline.ps1`，并把项目专属 rule/skill 纳入提交包，便于评审复现 workflow。

## 7. 决策收益与鲁棒性补强

- 结果：新增 `evaluate_decision_benefits.py`，输出当前控制、传统保守定值、噪声、设备波动、传感器平直段和响应时间评估。
- 风险：传统保守定值采用训练期 P90 作为人工定值代理，仍不是现场人工 A/B 试验。
- 改进：报告中同时披露当前控制对照和传统定值对照，避免只用单一基线放大收益。

## 8. 评审索引与论文依据表

- 结果：新增 `final_scoring_matrix.md` 和 `literature_basis_table.md`，把赛题 20/50/30/20 评分项、论文依据和项目创新逐项映射到仓库证据。
- 风险：证据索引会随指标文件更新而变旧，需要在重新训练或重新评估后同步检查。
- 改进：将两份索引加入提交包，并在 README 与完成度审计中指向它们。
